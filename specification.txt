OpenIDE Software Specification

1. Overview
OpenIDE is an IDE for FPGA engineers to visually connect VHDL modules, enabling the creation of structural FPGA designs. 
The application provides a graphical interface for adding modules, wiring signals, and managing project files. The user opens a software window to add VHDL modules and wire them together.

2. Functional Requirements
2.1 Module Management
	- Users can add, remove, and configure VHDL modules.
	- Each module is displayed as a green block with its name and ports (in/out, type).
	- Module coordinates are integer only.
2.2 Grid
	- Each position that is a multiple of 100 displays a small, furtive dot acting as a snap grid.
	- The grid is dynamically generated based on the current viewport (only visible grid points are drawn) to maintain high performance.
2.3 Wiring & Signals
	- Each wire between modules represents a signal (std_logic, std_logic_vector, etc.).
	- Ports are visualized as blue arrows (input: pointing to the wire, output: pointing out of the wire).
	- Wires are drawn manually: the user clicks to start, then draws lines with 90° angles.
	- Each wire is saved as a list of integer coordinates to retrace the whole wire.
	- Wires can only be placed at positions that are multiples of 100.
	- Wires are not automatically connected; user interaction is required.
2.4 VHDL Block (Module)
	- Each block has a library attribute, defaulted to "work".
	- Each block has an entity name inherited from the VHDL entity declaration.
	- Each block has a user-defined instance name, defaulted to "U_N" where N is the lowest integer (starting from 0) not already used in the design.
	- Each block has ports with direction: "in", "out", "inout", or "buffer". Port definitions are inherited from the VHDL entity.
	- Port placement defaults to the left side for inputs, and the right side for all other directions.
	- Ports can be freely repositioned to the left, right, top, or bottom side of the block.
	- "in" and "out" ports are drawn as small arrowhead-only markers (no tail line). The arrowhead points toward the block for "in", and away from the block for "out".
	- "inout" and "buffer" ports are drawn as small diamond markers.
2.5 Menus
	- 1st menu: Open a project, options, and space for future features.
	- 2nd menu: Draw a structural designer.
	- 3rd menu: Reserved/empty for future expansion.

2.6 Tool-Bar & Object Selection
	- A tool-bar is present at the top of the window, allowing the user to:
		-> Open project
		-> Save project
		-> Select object
		-> Draw wire
		-> Import VHDL module
		-> Optimal recenter
	- Each tool-bar option is displayed as a small PNG icon only (no text label on the button).
	- When the mouse hovers over an icon for 0.5s, a tooltip appears explaining the icon's function.
	- When 'Save project' is chosen:
		- If no project file path is known yet, open a file dialog so the user can choose where to save.
		- If a project file path is already known, save directly to that file without showing a dialog.
		- After a successful save, display a temporary notification in the top-right corner of the window that reads "Saved" and automatically disappears after 1 second.
	- When 'Select object' is chosen, the user can:
		- Click and drag on empty space to draw a selection rectangle. On release, every object (module or wire) that is entirely or partially inside the rectangle becomes selected.
		- Hold CTRL and left-click an object to toggle it in/out of the current selection (additive selection).
		- Click and drag a wire segment (not on a node) to move the entire wire.
		- Click and drag a wire node to move that single node.
		- Click and drag a VHDL block to move it.
		- When multiple objects are selected, dragging any one of them moves all selected objects together.
	- When 'Import VHDL module' is chosen, a file browser opens at the last-used directory (initially the launch directory). The user can select one or multiple VHDL files to import.
		- The last selected directory is persisted so subsequent imports reopen the same location.
	- When 'Optimal recenter' is chosen, the viewport position and zoom are adjusted to fit all modules and wires in the design within the visible area.

3. Technical Requirements
3.1 Drawing & Performance
	- All graphics are vector-based for high performance and scalability.
	- Drawing format is optimized for vector operations.
	- User can zoom and unzoom the workspace for detailed or overview editing.
	- Zoom and unzoom are centered on the mouse cursor position, so the point under the cursor stays fixed.
	- Zoom-in is capped at a maximum factor of 10.0; there is no limit on zoom-out.
	- Object hit-testing (clicking to select or drag)
		- Uses the exact cursor position, not the snapped grid position, so that selection accuracy is independent of the grid spacing.
		- Rdius is defined in screen pixels and scale with the zoom level, ensuring consistent clickable areas at any zoom.
	- When an object is selected, its visual appearance changes to indicate selection:
		- Wires (signals): highlighted in blue with a slightly thicker line and larger label font.
		- VHDL blocks: the module boundary is drawn in blue.

3.2 Action Remembering (Undo / Redo)
	- Every user action that modifies the design is recorded in an action history.
	- Recorded actions include (but are not limited to):
		- Moving one or more elements from position A to position B.
		- Deleting one or more elements.
		- Adding one or more elements.
	- CTRL+Z undoes the most recent action (repeatable to undo multiple steps).
	- CTRL+E redoes the last undone action (repeatable to redo multiple steps).
	- The redo history is cleared whenever a new action is performed after an undo.

3.3 File Format & Git Compatibility
	- All saved files are text-based, human-readable, and optimized for Git version control (easy to merge, track, and understand).
	- File formats are designed for easy commit, merge, and review.
3.4 Code Architecture
	- The codebase must be modular, partitioned, and non-monolithic to ensure flexibility and upgradability.
	- Clear separation of concerns between UI, logic, and data handling.
	- Partitioned code allows independent upgrades and changes.

4. Logging & Debugging
	- Comprehensive logging of software behavior and user actions.
	- Logs must facilitate debugging and traceability.
	- Every user input and software event is tracked in logs for high debuggability.

5. Extensibility & Upgradability
	- The application must be designed for easy upgrades and feature additions.
	- Partitioned code structure to allow independent module updates.
	- The code must not be monolithic, ensuring long-term flexibility.

6. User Experience
	- Intuitive interface for module placement, wiring, and project management.
	- Visual clarity and responsiveness prioritized.
	- User interaction for drawing wires and placing modules is clear and direct.
	- User can move the workspace view using arrow keys for navigation.
	- Pressing Escape cancels any ongoing action (drawing, moving, etc).
	- Pressing CTRL+Z reverses actions (undo) as many times as possible during the session.
	- Pressing CTRL+S runs "Save Project".
	- Pressing SUPPR (Delete) deletes all currently selected objects (VHDL block, wires, etc...)

7. Future Considerations
	- Menu and feature placeholders for future growth.
	- Maintain documentation and code comments for maintainability.
